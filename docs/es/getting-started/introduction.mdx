Los agentes de IA se ejecutan con cualquier privilegio que les concedas, y la mayoría de las veces son demasiados. Pueden ver claves de API en el entorno, acceder libremente a la red (incluida la metadata de la nube en `169.254.169.254`), y nada impide que una inyección de prompt ejecute `rm -rf /` en el sistema de archivos del host. Los contenedores ayudan con parte de esto, pero comparten el kernel del host, y los escapes de namespace son una clase de vulnerabilidad bien documentada.

microsandbox adopta un enfoque diferente: cada sandbox es una VM real con su propio kernel Linux. Proporciona primitivas de seguridad para evitar exploits como la exfiltración de secretos. Y **se ejecuta localmente** en tu máquina.

<CodeGroup>

```rust Rust
use microsandbox::{Sandbox, NetworkPolicy};

let sb = Sandbox::builder("sandbox")
    .image("python")
    .memory(512)
    .cpus(2)
    .secret(|s| s
        .env("OPENAI_API_KEY")
        .value(std::env::var("OPENAI_API_KEY")?)
        .allow_host("api.openai.com")
    )
    .network(|n| n.policy(NetworkPolicy::public_only()))
    .create()
    .await?;
```

```typescript TypeScript
import { NetworkPolicy, Sandbox } from "microsandbox";

await using sb = await Sandbox.builder("sandbox")
    .image("python")
    .memory(512)
    .cpus(2)
    .secret((s) =>
        s.env("OPENAI_API_KEY")
            .value(process.env.OPENAI_API_KEY!)
            .allowHost("api.openai.com"),
    )
    .network((n) => n.policy(NetworkPolicy.publicOnly()))
    .create();
```

```python Python
import os
from microsandbox import Network, Sandbox, Secret

sb = await Sandbox.create(
    "sandbox",
    image="python",
    memory=512,
    cpus=2,
    secrets=[
        Secret.env("OPENAI_API_KEY",
            value=os.environ["OPENAI_API_KEY"],
            allow_hosts=["api.openai.com"],
        ),
    ],
    network=Network.public_only(),
)
```

</CodeGroup>

## Lo básico

- **Arranque en menos de 100 ms.** Los sandboxes arrancan en menos de 100 milisegundos.
- **Sin daemon.** El runtime se ejecuta directamente como un proceso hijo de la aplicación que crea el sandbox. Sin root, sin servidor, sin servicio en segundo plano.
- **Cualquier imagen OCI.** Docker Hub, GHCR, ECR, GCR o cualquier registro compatible con OCI. Las capas compartidas se deduplican y solo se almacenan una vez.
- **Multiplataforma.** Nativo en macOS (Apple Silicon) y Linux (KVM). El mismo CLI, los mismos SDK.
- **SDK multilenguaje.** Rust, TypeScript y Python, todos con APIs consistentes.
- **Programable.** Las políticas de red, hooks de sistema de archivos, vinculación de secretos e interceptación de TLS se configuran desde el lado del host a través del SDK.

## Qué lo hace diferente

### Secretos que no pueden filtrarse

La mayoría de los sandboxes protegen los secretos restringiendo lo que el invitado puede hacer. microsandbox va más allá: **el invitado nunca tiene el secreto en absoluto**.

En lugar de inyectar credenciales reales en el sandbox, microsandbox genera marcadores de posición aleatorios. El valor real nunca entra en la VM. La única forma de que llegue al mundo exterior es cuando una solicitud va a un host permitido, momento en el cual microsandbox reemplaza el marcador por el valor real. En todos los demás lugares, el marcador es solo una cadena sin sentido.

Así, incluso con ejecución completa de código dentro del sandbox, no hay nada que exfiltrar. La credencial nunca estuvo allí. La protección contra DNS rebinding y el bloqueo de metadata de la nube se activan automáticamente cuando se configuran secretos, cerrando los canales laterales restantes.

<CodeGroup>

```rust Rust
let sb = Sandbox::builder("sandbox")
    .image("python")
    .secret_env("OPENAI_API_KEY", api_key, "api.openai.com")
    .create()
    .await?;
```

```typescript TypeScript
await using sb = await Sandbox.builder("sandbox")
    .image("python")
    .secretEnv("OPENAI_API_KEY", process.env.OPENAI_API_KEY!, "api.openai.com")
```

### Capa de red programable <sup><sup>próximamente</sup></sup>

Cada paquete que sale de un sandbox pasa por una pila de red controlada completamente desde el lado del host, donde se aplica la política a nivel de IP, DNS o HTTP. Desde dentro del sandbox parece una interfaz de red normal, pero en el lado del host tú decides qué pasa.

No hay una red host subyacente a la que conectarse, ni reglas que eludir, ni forma de que el invitado evite el filtro.

<CodeGroup>

```rust Rust
let sb = Sandbox::builder("sandbox")
    .image("python")
    .network(|n| n
        .policy(NetworkPolicy::allowlist(["api.openai.com", "pypi.org"]))
    )
    .create()
    .await?;
```

```typescript TypeScript
import { Destination, NetworkPolicy, Rule, Sandbox } from "microsandbox";

await using sb = await Sandbox.builder("sandbox")
    .image("python")
    .network((n) => n.policy({
        defaultEgress: "deny",
        defaultIngress: "allow",
        rules: [
            Rule.allowEgress(Destination.domain("api.openai.com")),
            Rule.allowEgress(Destination.domain("pypi.org")),
        ],
    }))
    .create();
```

```python Python
from microsandbox import Network, NetworkPolicy, Rule, Sandbox

sb = await Sandbox.create(
    "sandbox",
    image="python",
    network=Network(
        policy=NetworkPolicy(rules=(
            Rule.allow(destination="api.openai.com"),
            Rule.allow(destination="pypi.org"),
        )),
    ),
)
```

</CodeGroup>

### Backends de sistema de archivos extensibles <sup><sup>próximamente</sup></sup>

El host controla lo que el invitado ve a nivel de sistema de archivos. Monta directorios del host, usa volúmenes gestionados o conecta backends de sistema de archivos personalizados por completo. Los hooks te permiten interceptar lecturas y escrituras para hacer cosas como cifrar todo lo que va al disco o redirigir a almacenamiento remoto, y el invitado nunca nota la diferencia.

<CodeGroup>

```rust Rust
let sb = Sandbox::builder("encrypted")
    .image("python")
    .volume("/secrets", |v| v
        .bind("/data/secrets")
        .on_read(move |_path, data| decrypt(data, &key))
        .on_write(move |_path, data| encrypt(data, &key))
    )
    .create().await?;
```

```typescript TypeScript
// Los hooks de sistema de archivos (onRead / onWrite) llegan próximamente para Node. Por ahora,
// puedes montar el directorio del host directamente:
await using sb = await Sandbox.builder("encrypted")
    .image("python")
    .volume("/secrets", (m) => m.bind("/data/secrets"))
    .create();
```

```python Python
from microsandbox import Sandbox, Volume

key = get_encryption_key()
sb = await Sandbox.create(
    "encrypted",
    image="python",
    volumes={
        "/secrets": Volume.bind("/data/secrets",
            on_read=lambda path, data: decrypt(data, key),
            on_write=lambda path, data: encrypt(data, key),
        ),
    },
)
```

</CodeGroup>

### Captura y reutiliza el estado de configuración

Captura la capa superior escribible de un sandbox detenido como un artefacto portátil y luego arranca sandboxes nuevos a partir de ella. Instala dependencias una vez, haz un snapshot, y ejecuta `msb run --snapshot ...` para workers que se saltan la fase de configuración. El artefacto es un directorio que puedes copiar entre máquinas con `scp` o empaquetar en un `.tar.zst`. Los snapshots actualmente son solo de disco y solo de sandboxes detenidos; los snapshots de estado en vivo de la VM están planeados como trabajo futuro.

<CodeGroup>

```rust Rust
let sb = Sandbox::builder("baseline")
    .image("python")
    .create()
    .await?;

sb.exec("pip", ["install", "-r", "requirements.txt"]).await?;
sb.stop_and_wait().await?;

// Hacer snapshot del sandbox detenido con un nombre simple
let h = Sandbox::get("baseline").await?;
let _snap = h.snapshot("after-pip-install").await?;

// Arrancar 10 workers desde el mismo snapshot (cada uno con su propia capa superior)
let workers: Vec<Sandbox> = futures::future::try_join_all(
    (0..10).map(|i| async move {
        Sandbox::builder(format!("worker-{i}"))
            .from_snapshot("after-pip-install")
            .create()
            .await
    })
).await?;
```

```bash CLI
msb run --name baseline --detach python -- sleep 3600
msb exec baseline -- pip install -r requirements.txt
msb stop baseline

msb snapshot create after-pip-install --from baseline

# Arrancar 10 workers desde el snapshot
for i in $(seq 1 10); do
    msb run --name worker-$i --detach --snapshot after-pip-install -- sleep 3600
done
```

```typescript TypeScript
import { Sandbox } from "microsandbox";

const sb = await Sandbox.builder("baseline").image("python").create();
await sb.exec("pip", ["install", "-r", "requirements.txt"]);
await sb.stopAndWait();

const h = await Sandbox.get("baseline");
await h.snapshot("after-pip-install");

// Arrancar 10 workers desde el mismo snapshot
const workers = await Promise.all(
    Array.from({ length: 10 }, (_, i) =>
        Sandbox.builder(`worker-${i}`).fromSnapshot("after-pip-install").create(),
    ),
);
```

```python Python
import asyncio
from microsandbox import Sandbox

sb = await Sandbox.create("baseline", image="python")
await sb.exec("pip", ["install", "-r", "requirements.txt"])
await sb.stop_and_wait()

h = await Sandbox.get("baseline")
await h.snapshot("after-pip-install")

# Arrancar 10 workers desde el mismo snapshot
workers = await asyncio.gather(
    *(Sandbox.create(f"worker-{i}", snapshot="after-pip-install") for i in range(10))
)
```

</CodeGroup>

### Crea sandboxes desde sandboxes <sup><sup>próximamente</sup></sup>

El código que se ejecuta dentro de un sandbox puede crear sandboxes pares a su lado. Esto está diseñado para sistemas multiagente donde los agentes necesitan crear subentornos. Los sandboxes pares no heredan nada por defecto (su propia red, sistema de archivos, secretos), y el orquestador coordina a través del SDK, no a través de estado compartido. Si el código no se está ejecutando dentro de un microsandbox, la llamada de creación falla de forma segura.

### Sistema de plugins componibles <sup><sup>próximamente</sup></sup>

Extiende microsandbox con plugins de Rust en proceso o plugins fuera de proceso en cualquier lenguaje. Los plugins se enganchan en eventos del ciclo de vida, llamadas a exec, operaciones del sistema de archivos o tráfico de red. Se componen de forma natural, así que puedes apilar un registrador de auditoría, un limitador de tasa y un monitor de red sin que se conozcan entre sí.

### Eventos bidireccionales <sup><sup>próximamente</sup></sup>

Los procesos invitados pueden emitir eventos tipados al host, y el host puede enviar eventos de vuelta. No se requieren librerías especiales en el invitado, ya que cualquier lenguaje puede escribir JSON en un socket. Esto permite la comunicación estructurada más allá de stdin/stdout, como reportes de progreso, señales de finalización de tareas o RPC personalizado.

<CodeGroup>

```rust Rust
// Recibir eventos del invitado
let _sub = sb.on_event("task.progress", |event: EventData| {
    if let Ok(p) = event.data.unwrap().deserialized::<Progress>() {
        println!("[{}%] {}", p.percent, p.message);
    }
});

// Enviar eventos al invitado
sb.emit("task.start", TaskConfig {
    input_file: "/data/input.txt".into(),
    output_dir: "/data/output".into(),
}).await?;
```

```typescript TypeScript
// Recibir eventos del invitado
const sub = sb.onEvent("task.progress", (event: EventData) => {
    const { percent, message } = event.data as { percent: number; message: string }
    console.log(`[${percent}%] ${message}`)
})

// Enviar eventos al invitado
await sb.emit("task.start", { inputFile: "/data/input.txt", options: ["--verbose"] })
```

```python Python
# Recibir eventos del invitado
sub = sb.on_event("task.progress", lambda event: (
    print(f"[{event.data['percent']}%] {event.data['message']}")
))

# Enviar eventos al invitado
await sb.emit("task.start", {
    "input_file": "/data/input.txt",
    "output_dir": "/data/output",
})
```
